Chapter 02

البنية المعمارية لنظام لينكس بالتفصيل

Linux Architecture in Details

يعتمد هذا الدرس على المفاهيم التي تم شرحها في الدرس الأول (POSIX، GNU، Glibc، Kernel) وينتقل إلى فهم أعمق لكيفية تواصل هذه المكونات مع بعضها ومع العتاد.

01

البنية المعمارية لنظام لينكس Linux Architecture

ينقسم نظام لينكس بشكل أساسي إلى ثلاث طبقات رئيسية:

Linux System Architecture
مساحة المستخدم User Space
التطبيقات العادية تعمل فيه (مثل المتصفح، سطر الأوامر، إلخ)
مساحة النواة Kernel Space
النواة Kernel تعمل فيه وتتمتع بصلاحيات مطلقة
العتاد Hardware
المكونات الفيزيائية مثل المعالج CPU، الذاكرة RAM، وحدات التخزين، والمنافذ
  1. مساحة المستخدم (User Space): المكان الذي تعمل فيه التطبيقات العادية للمستخدم (مثل المتصفح، سطر الأوامر، إلخ).
  2. مساحة النواة (Kernel Space): المكان الذي تعمل فيه النواة Kernel وتتمتع فيه بصلاحيات مطلقة.
  3. العتاد (Hardware): المكونات الفيزيائية مثل المعالج CPU، الذاكرة RAM، وحدات التخزين، والمنافذ.
لماذا تم فصل User Space عن Kernel Space؟؟
  • دواعي أمنية (Security Reasons): لو امتلكت تطبيقات المستخدم (أي تطبيق) صلاحية الوصول المباشر للعتاد، يمكن لفيروس/مخترق بالعبث بالذاكرة أو بالتحكم بالعتاد بشكل يضر بالنظام بالكامل.
  • تنظيم العمل: ترك مهمة إدارة الموارد والعمليات (Process Management / Scheduler) للنواة فقط لضمان عمل النظام باستقرار.
تتذكر؟ ماذا يجب دائماً تتكون البنية الأساسية للينكس من 3 أجزاء: User Space، Kernel Space، Hardware.
أخطاء شائعة الاعتقاد بأن التطبيقات التي نكتبها تتحكم مباشرة في الـ Hardware — هذا مستحيل ويمنع النظام من الجذور.
02

مستويات الصلاحيات في العتاد Hardware Exception Levels

الفصل بين مساحة النواة ومساحة المستخدم ليس مجرد "كود برمجي" بل هو مدعوم من قبل العتاد نفسه. يتم تنظيم هذا الفصل عبر ما يسمى بـ Exception Levels (EL) مع معالجات ARM (مثل Cortex-A).

تعريفات سريعة؟؟
تعريف ISA (Instruction Set Architecture): "القانون" أو التصميم الهندسي الذي يحدد شكل الأوامر (Instructions) والسجلات (Registers) الخاصة بالمعالج (مثل ARM v7, ARM v8).
تعريف Hypervisor: نظام أو برنامج يملك صلاحيات أعلى من الـ Kernel، وظيفته تشغيل أنظمة تشغيل عدة (Operating Systems) في نفس الوقت على نفس العتاد.
كيف تعمل مستويات الصلاحيات في معمارية ARM؟ ⚙

تُقسم الصلاحيات إلى 4 مستويات (من الأقل للأعلى):

ARM Exception Levels
EL3
Secure Monitor
أعلى مستوى صلاحيات. يتحكم في الجوانب الأمنية المتقدمة (TrustZone — سيتم شرحه في دروس متقدمة).
EL2
Hypervisor
يملك صلاحيات أعلى من النواة. يتحكم في تقسيم العتاد بين أنظمة تشغيل عدة تعمل بالتوازي.
EL1
Kernel Space
النواة تعمل هنا (OS Kernel). تملك صلاحية التحكم بالعتاد والـ Hardware Peripherals.
EL0
User Space
تعمل هنا تطبيقات المستخدم (Applications). مستوى صلاحيات أقل. لا تملك أي قدرة على الوصول المباشر للعتاد.
المستوى الاسم التقني الوظیفة والمكان العمل
EL0 User Space تطبيقات المستخدم Applications مستوى صلاحيات أقل. لا تملك أي قدرة على الوصول المباشر للعتاد.
EL1 Kernel Space النواة تعمل هنا OS Kernel تملك صلاحية التحكم بالعتاد والـ Hardware Peripherals.
EL2 Hypervisor يتحكم في تقسيم العتاد بين أنظمة تشغيل عدة تعمل بالتوازي يملك صلاحيات أعلى من النواة.
EL3 Secure Monitor يتحكم في الجوانب الأمنية المتقدمة (TrustZone — سيتم شرحه في دروس متقدمة) أعلى مستوى صلاحيات.
مقارنة أنواع الـ Hypervisor؟؟
وجہ المقارنة Type 1 Hypervisor (Bare-metal) Type 2 Hypervisor (Hosted)
كيف یعمل؟ یعمل مباشرة فوق العتاد Hardware. یعمل كتطبیق فوق نظام تشغیل موجود مسبقاً.
أین یُستخدم؟ الخوادم Servers، أنظمة السیارات Automotive. الأجھزة الشخصیة، الحواسیب العادیة.
أمثلة QNX (أحیاناً یُستخدم كـ Type 1)، Xen. VirtualBox، VMware، QEMU.
تتذكر؟ ماذا يجب إذا جاء الأمر من EL0 فإن المعالج CPU نفسه یعرف مصدر الأمر (Instruction) ویحاول الوصول إلى عنوان ذاكرة مخصص لـ EL1 فإن المعالج سیرفض التنفیذ فوراً للحمایة.
مراجعة سريعة EL0: User  |  EL1: Kernel  |  EL2: Hypervisor  |  EL3: Secure Monitor
03

كيف يتواصل User Space مع Kernel Space — الأهم: اللغز

فهم عميق هذا السؤال هو "فخ" شهير في المقابلات الوظیفية (Interviews). إذا كان التطبیق في User Space ولا یملك أي صلاحیات للوصول إلى Kernel Space أو العتاد، فكیف یستطیع مثلاً قراءة ملف من الھارد دیسك؟!
مثال توضیحي (مثال المواطن والإدارة)؟؟

المدرب استخدم هذا المثال المطابق للواقع التقني لتبسیط هذا اللغز المعقد:

  • المواطن العادي: هو التطبیق (Application / Binary File).
  • مكتب المدیر (والمجال بھ): هو الـ Kernel Space.
  • المدیر: هو الـ Kernel / Scheduler.
  • موظف داخل مكتب المدیر: هو العملیة (Process).

المواطن (Application) لا يمكنه الدخول لمكتب المدیر. الحل؟ أن هذا المواطن يتحول إلى "موظف" (Process) في مكتب المدیر. لكنه يبقى مواطن، الأصل يكتسب هویة جدیدة تسمح لھ بالعمل داخل حدود النظام وتقدیم الطلبات للمدیر.

كیف یعمل تحویل التطبیق إلى Process؟ ⚙
Process Creation Flow
الكود المصدري للتطبیق Binary File موجود في وحدة التخزین Storage
عند طلب تشغیل التطبیق، لا یتم نقل كوده بالكامل إلى Kernel Space لكي لا یكتسب صلاحیات مطلقة
بدلاً من ذلك، تقوم النواة بنسخھ إلى الذاكرة في User Space وتنشئ لھ ملف تعریف / هویة داخل Kernel Space (أو Task) یسمى Process
الـ Process یحتوي على بیانات التطبیق، مساحته في الذاكرة، ومعرّف خاص بھ ID
یقوم الـ Scheduler بإدارة الـ Processes ھذه وتحدیث من یعمل أولاً
  1. الكود المصدري للتطبیق Binary File موجود في وحدة التخزین Storage.
  2. عند طلب تشغیل التطبیق، لا یتم نقل كوده بالكامل إلى Kernel Space لكي لا یكتسب صلاحیات مطلقة.
  3. بدلاً من ذلك، تقوم النواة بنسخھ إلى الذاكرة في User Space وتنشئ لھ ملف تعریف / هویة داخل Kernel Space (أو Task) یسمى Process.
  4. الـ Process یحتوي على بیانات التطبیق، مساحته في الذاكرة، ومعرّف خاص بھ ID.
  5. یقوم الـ Scheduler بإدارة الـ Processes ھذه وتحدیث من یعمل أولاً.
تتذكر؟ ماذا يجب لیعمل أي تطبیق مباشرة یجب أن تقوم النواة Kernel بتحویلھ أولاً إلى عملیة Process قابلة للجدولة (Scheduling).
أخطاء شائعة الاعتقاد بأنه عندما یعمل التطبیق فإنه یتم نقلھ إلى الكیرنل. الصحیح أن الكیرنل ینشئ لھ فقط "هویة/ملف" Process لیتبعھ، ویبقى التطبیق معزولاً في User Space فقط.
04

نداءات النظام System Calls — Syscalls

حتی بعد أن یصبح التطبیق Process ویعمل، فھو لا یزال لا یمتلك وصولاً للعتاد. يجب علیھ أن "یطلب" ذلك من الكیرنل عبر آلیة مقدسة تسمى System Call نداء النظام.

تعريف System Call: هو طلب رسمي ووحید یقوم بھ التطبیق في User Space لیطلب من النواة Kernel Space أداء مھمة تحتاج لصلاحیات (مثل فتح ملف، القراءة، الكتابة).
كیف تعمل الـ System Calls فعلیاً (بخطوة)؟ ⚙

التطبیق لا یمكنھ التحدث مباشرة للكیرنل. وسیلتھ الوحیدة هی بما یسمى بالمقاطعات (Interrupts / الاستثناءات) لفت انتباه المعالج CPU.

System Call Execution Flow
1
تجھیز الطلب: كل نظام نداء لھ رقم تعریفي ID ومعاملات Arguments (مثلاً: نداء الكتابة write لھ رقم 1 في بعض المعماریات).
2
استخدام السجلات CPU Registers: یقوم التطبیق بوضع الـ ID الخاص بالنداء في معین سجل في الـ CPU (مثلاً سجل X8 في معماریة ARM، أو RAX في معماریة x86_64) ویضع المعاملات Arguments في سجلات أخرى.
3
إطلاق المقاطعة Trigger Interrupt: یقوم التطبیق بتنفیذ أمر Assembly خاص (مثل svc في ARM أو syscall في x86).
4
توقف المعالج وتنبیھ النواة: هذا الأمر یوقف عمل الـ CPU العادی ویطلق Interrupt ویخبر الـ Kernel أنھ هناك انتباهاً مطلوباً.
5
تنفیذ الطلب: الـ Kernel یتفحص السجلات Registers یقرأ الـ ID (یتعرف مثلاً أنھ أمر write) ینفذ المھمة ثم یعید النتیجة للتطبیق.
العلاقة بین مكتبة C Library والـ System Calls؟؟
  • دوال لغة C الموجودة في مكتبة glibc مثل printf أو scanf هي في الحقیقة مجرد واجھة مبسطة (Abstraction).
  • دالة printf من الداخل تقوم بتجھیز السجلات Registers وتستدعی نداء النظام الحقیقي وھو write.
  • يمكنك التخلي تماماً عن glibc والقیام بـ System Call مباشرة بكتابة أوامر Assembly (ولكن هذا صعب وغیر عملي برمجیاً).
Assembly (ARM64) 
// مثال: نداء نظام write مباشرة عبر Assembly على ARM64

MOV X0, #1       // fd = 1 (stdout)
ADR X1, msg       // عنوان النص
MOV X2, #13      // طول النص
MOV X8, #64      // syscall number لـ write في ARM64
SVC #0             // إطلاق المقاطعة — الاستدعاء

msg: .ascii "Hello, World!\n"
Assembly (x86_64) 
// مثال: نداء نظام write مباشرة عبر Assembly على x86_64

MOV RAX, 1        // syscall number لـ write = 1
MOV RDI, 1        // fd = 1 (stdout)
MOV RSI, msg       // عنوان النص
MOV RDX, 13       // طول النص
SYSCALL              // إطلاق المقاطعة على x86

msg: .ascii "Hello, World!\n"
printf to System Call Flow
printf() — دالة في مكتبة C Library (glibc)
تجھیز السجلات Registers — وضع ID والمعاملات Arguments
تنفیذ أمر المقاطعة (svc / syscall)
write() — System Call الحقیقي في النواة
تتذكر؟ ماذا يجب التطبیق لا یصل للعتاد بل یضع أوامره في الـ Registers ثم یقوم بعمل Interrupt لیجبر الـ Kernel على الاستجابة لطلبھ.
مراجعة سريعة printf ⬅ System Interface (C Library)  |  write ⬅ System Call
05

واجهة التطبیقات الثنائیة ABI — Application Binary Interface

كيف یعرف التطبیق أن رقم نداء write هو 1 وما یضمن أن الكیرنل سیعتبر الرقم 1 هو write ولیس read؟

تعريف ABI (Application Binary Interface): هو قانون أو معیار مقدس وثابت داخل نظام لینكس. یتكون من جدول یحدد بدقة متناھیة:
  1. أرقام IDs الخاصة بكل System Call.
  2. طریقة استدعاء الدوال (Calling Convention) وأسماء السجلات Registers التی یجب أن توضع فیھا الـ IDs والمعاملات.
لماذا یعتبر الـ ABI ثابتاً ومقدساً؟؟
مهم لو قرر مطورو نواة لینكس (Linus Torvalds وفریقھ) تغییر رقم الـ write في جدول الـ ABI من 1 إلى 50 مثلاً... فإن جمیع البرامج الحالیة ستتوقف عن العمل فوراً. ذلك یعني تغییر الـ ABI بالكامل. التطبیقات القدیمة الموجودة في العالم والتي تم عملها (Compile) مسبقاً على الرقم 1 ستنھار مثلاً. لذلك، الـ ABI في لینكس لا یتغیر أبداً (Stable).
API vs ABI
API — واجهة الكود المكتوب (المبرمج) مثل printf
ABI — واجهة النظام للتعامل مع التطبیقات بعد أن تتحول إلى Binary (كیفیة ترتیب الـ Registers وأرقام الـ Syscalls)
تتذكر؟ ماذا يجب API مثل printf هي واجهة الكود المكتوب (المبرمج). ABI هي واجهة النظام للتعامل مع التطبیقات بعد أن تتحول إلى Binary (كیفیة ترتیب الـ Registers وأرقام الـ Syscalls).
06

ما هو الشل Shell؟ وكیف تبدأ التطبیقات؟

إذا كان تشغیل التطبیقات یحتاج إلى الدخول إلى الكیرنل لإنشاء Process، فكیف نقوم بتشغیل أول تطبیق؟

كیف یشتغل التطبیق؟؟

في الواقع یجب أن یقوم آخر تطبیق یعمل فعلاً بعمل System Call مثل execve لیطلب من الكیرنل تشغیل تطبیق معین.

ما هو دور الـ Shell؟ ⚙
  1. الشل هو مجرد تطبیق Application/Process یعیش في User Space مثل أي تطبیق آخر.
  2. وظیفته الأساسیة: هو "الواسطة" التي نستخدمھا لطلب تشغیل التطبیقات الأخرى من الكیرنل.
  3. یمتلك كلغة برمجة خاصة بھ (Programming Language) كمترجم تحتوي على شروط وحلقات تكرار (if، while، for).
  4. أمثلة على الشل: bash، sh، zsh (جمیعھا Implementations مختلفة لنفس مفھوم الـ POSIX Shell).
07

مدير الإقلاع الأول Init Manager / Init Program

إذا كان كل تطبیق یحتاج إلى تطبیق قبله لتشغیلھ... فمن یشغل الأول في النظام؟!

تعريف init: هو أول Process یعمل في نظام لینكس إطلاقاً (ویكون رقمه ID دائماً هو 1).
كیف یعمل؟ ⚙
  1. عندما یكتمل تحمیل الكیرنل في الذاكرة ویتعرف على العتاد.
  2. الكیرنل یبحث في وحدة التخزین File System عن تطبیق یسمى init.
  3. یقوم الكیرنل بتشغیل ھذا التطبیق يدویاً من داخلھ.
  4. تطبیق init یأخذ عاتقھ مسؤولیة تشغیل كافة التطبیقات والخدمات Services الأخرى في النظام (بما فیھا Shell الشل).
Linux Boot Sequence (Simplified)
العتاد Hardware — التشغیل الفيزیائي
الكیرنل Kernel — یحمّل نفسھ في الذاكرة
init (PID 1) — أول Process في النظام
الخدمات والتطبیقات Services & Applications — بما فیھا Shell
مقارنة تطبیقات إدارة الإقلاع Init Managers Implementations؟؟

بما أن لینكس نظام مفتوح المصدر، هناك عدة تطبیقات بُنیت لأداء دور الـ init (تذكر: الكونسبت واحد والتطبیقات مختلفة):

— SysVinit systemd
النوع النظام القدیم والكلاسیكي (Sequential) النظام الحدیث والأشھر حالیاً
كیف یشغل الخدمات؟ یشغل الخدمات واحدة تلو الأخرى بالترتیب یشغل الخدمات بالتوازي (Parallel) مما یسرع الإقلاع
الخصائص بسیط، خفیف، ومعقد قوي، وأصبح المعیار الافتراضي في توزیعات Ubuntu و Fedora و Yocto
تتذكر؟ ماذا يجب النواة تشغل init ثم init یشغل الخدمات والتطبیقات. الـ Init Manager لمفھوم init هو مجرد Implementation مثل systemd.
08

نصائح وأسئلة شائعة Q&A

Q كیف یكتب التطبیق في سجلات المعالج CPU Registers إذا كان لا یملك صلاحیات وصول للعتاد؟
التطبیق لا یلمس المعالج بیده. الكیرنل یوجه الـ CPU لجلب الأوامر (Fetch) من مساحة ذاكرة التطبیق في User Space. المعالج ینفذ الأوامر العادیة (مثل وضع قیمة في Register) طالما أنھا لا تخالف مستواه الأمني (EL0) — التطبیق یقتحم من المعالج ولیس العكس.
Q هل یھم لغة البرمجة C, C++, Rust في Embedded Linux؟
نعم جداً:
  • C: الأھم والأقوى على مستوى النظام (System Level).
  • C++: قویة جداً في تطبیقات User Space الثقیلة.
  • Rust: بدأت تكتسب قوة ھائلة حتى داخل الكیرنل بسبب میزات الأمان القویة التي تمنع أخطاء الذاكرة.
Q كتب یُنصح بھا بشدة؟؟

النصیحة الذھبیة من المدرب

استوعبوا المفاھیم الأساسیات والفلسفیة للأنظمة ببطء وصبركم. لا تتركوا الذكاء الاصطناعي یأخذ طاقة من التعلم، لأن ھذه الأساسیات (مثل الـ Architecture والـ System calls) یفھم منھا فیسرع سیطرة بشدة في المراحل المتقدمة (مثل كتابة الـ Drivers).

مراجعة سريعة للدرس

المفاهیم الرئيسیة
  • لینكس ینقسم إلى: User Space، Kernel Space، Hardware
  • الفصل بین المساحتین مدعوم عتادیاً عبر Exception Levels
  • التطبیق لا یعمل مباشرة بل یتحول لـ Process
  • التواصل بین المساحتین یتم عبر System Calls
  • الـ ABI هو المعیار الثابت الذي یضمن توافق التطبیقات
  • init (PID 1) هو أول عملیة في النظام
تعریفات مهمة
  • ISA: التصمیم الهندسي للأوامر والسجلات
  • Hypervisor: برنامج یشغل أنظمة تشغیل عدة على نفس العتاد
  • System Call: طلب رسمي من التطبیق للنواة
  • ABI: قانون ثابت یحدد أرقام الـ Syscalls وطریقة الاستدعاء
  • Shell: وسیطة بین المستخدم والكیرنل
  • init: أول Process في النظام (PID 1)
أخطاء شائعة
  • الاعتقاد أن التطبیقات تتحكم مباشرة في Hardware
  • الاعتقاد أن التطبیق ینقل للكیرنل عند التشغیل (الصحیح: ینشئ لھ Process في User Space)
  • خلط مفھوم API مع ABI
  • اعتبار أن systemd هو الكونسبت نفسه (ھو مجرد Implementation)
ملاحظات المقابلات
  • "كیف یتواصل User Space مع Kernel Space؟" — سؤال فخ شھیر
  • الجواب: عبر System Calls بعد تحویل التطبیق لـ Process
  • قد یُسأل عن الفرق بین Type 1 و Type 2 Hypervisor
  • قد یُسأل عن شرح خطوات System Call بالتفصیل
نصائح للحفظ
  • EL0=User، EL1=Kernel، EL2=Hypervisor، EL3=Secure
  • printf ← واجهة مبسطة ← write ← System Call حقیقي
  • العتاد → الكیرنل → init → الخدمات → Shell → التطبیقات
  • مواطن + مكتب المدیر = فهم User Space vs Kernel Space
نصائح الامتحان
  • ركز على شرح سبب فصل User Space عن Kernel Space (الأمان والاستقرار)
  • احفظ جدول Exception Levels بالأرقام والأسماء
  • اكتب خطوات System Call مرتبة (1-5)
  • اذكر أن ABI لا یتغیر أبداً ولماذا
قائمة المراجعة النهائیة
  • أستطیع شرح البنیة الثلاثیة للینكس (User Space، Kernel Space، Hardware)
  • أعرف لماذا تم الفصل بین المساحتین (أمنیة + تنظیم)
  • أستطیع شرح Exception Levels الأربعة في ARM
  • أستطیع المقارنة بین Type 1 و Type 2 Hypervisor
  • أعرف كيف یتحول التطبیق إلى Process (مثال المواطن)
  • أستطیع شرح خطوات System Call الخمسة بالتفصیل
  • أفهم الفرق بین printf و write
  • أعرف ما هو ABI ولماذا هو ثابت ومقدس
  • أفهم دور Shell في النظام
  • أعرف تسلسل الإقلاع: العتاد → الكیرنل → init → الخدمات
  • أستطیع المقارنة بین SysVinit و systemd
