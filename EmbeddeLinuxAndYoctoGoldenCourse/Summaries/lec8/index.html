1 فلسفة الترجمة The Compilation Philosophy

لماذا نحتاج لعملية الترجمة Compilation؟

المعالج CPU الموجود في أي حاسوب أو لوحة Board لا يفهم لغات البرمجة عالية المستوى مثل C أو C++، فهو مصمم هندسياً Architecture لفهم لغة الآلة فقط Machine Code - Binary 0s and 1s.

عملية الترجمة هي تحويل الكود المصدري المقروء للبشر إلى لغة يفهمها العتاد لتنفيذها.

جدول مقارنة: اللغات المترجمة مقابل اللغات المفسرة

وجه المقارنة Compiled Languages Interpreted Languages
الآلية يتم تحويل الكود بالكامل إلى ملف تنفيذي Binary قبل التشغيل يتم قراءة الكود سطراً بسطراً وتنفيذه في وقت التشغيل
الأداء سريعة جداً أبطأ جداً لأنها تحتاج إلى مترجم/مفسر Interpreter يترجم في نفس اللحظة
أمثلة C, C++, Rust Shell, Python, JavaScript

الأداة السحرية gcc ليست مترجماً واحداً!

أغلب المهندسين يعتقدون أن أمر gcc هو المترجم، وهذا غير دقيق. gcc (GNU Compiler Collection) عبارة عن سائق أو مدير Driver، وظيفته استدعاء 4 برامج أخرى تعمل في الخفاء بالتسلسل لتتم عملية البناء بشكل آلي.

✅
ماذا يجب أن تتذكر؟ — أداة gcc مجرد غلاف Wrapper ينظم استدعاء أدوات أصغر للقيام بالمهمة. المعالج لا ينفذ لغة C، بل ينفذ تعليمات ثنائية Binary.
⚠️ أخطاء شائعة

الخلط بين اللغات المترجمة Compiled ولغات المفسرة Interpreted كالشل Shell الذي درسناه سابقاً. اللغات المفسرة لا تمر بهذه المراحل، بل الكود ينفذ مباشرة عبر Interpreter.

2 مراحل الترجمة الأربعة The 4 Stages of Compilation

عملية البناء لا تتم بخطوة واحدة، بل تمر بـ 4 مراحل دقيقة ومتتالية Workflow:

Compilation Pipeline — خط أنابيب الترجمة
Source Code .c
1. Preprocessor (cpp)
Expanded Code .i
2. Compiler (cc1)
Assembly Code .s
3. Assembler (as)
Object File .o
4. Linker (ld)
Executable Binary

المرحلة الأولى: المعالجة المسبقة Preprocessing

الأداة المستخدمة: cpp (C Preprocessor)

الوظيفة: تجهيز ملف C قبل الترجمة الفعلية. يقوم البرنامج بـ:

  • مسح جميع التعليقات Comments لأن المعالج لا يحتاجها وهي فقط تستهلك مساحة
  • توسيع الماكرو Macro Expansion: تعويض كل #define بقيمته الحقيقية في الكود
  • إدراج الملفات File Inclusion: نسخ محتوى ملفات الـ Header (مثل #include <stdio.h>) بالكامل ولصقه داخل الكود

الملف الناتج: ملف كود C نقي وضخم جداً، امتداده الافتراضي .i

المرحلة الثانية: الترجمة الفعلية Compilation / Translation

الأداة المستخدمة: cc1 (لبرامج C) أو cc1plus (لبرامج C++)

الوظيفة: أخذ الملف الناتج من المرحلة الأولى وتحويله من لغة C إلى لغة التجميع Assembly Language.

🧠 فهم عميق

ملاحظة هامة: لغة Assembly مرتبطة جداً بالهاردوير، فهي تختلف باختلاف معمارية المعالج (x86 تختلف عن ARM).

الملف الناتج: ملف نصي يحتوي على أوامر Assembly، امتداده .s

المرحلة الثالثة: التجميع Assembly

الأداة المستخدمة: as (Assembler)

الوظيفة: تحويل ملف Assembly النصي إلى لغة الآلة Machine Code (أصفار وآحاد).

الملف الناتج: ملف يُسمى Object File بامتداد .o. لم يعد هذا الملف قابلاً للقراءة كنص، ولكنه يعد قابلاً للتشغيل لاحقاً.

المرحلة الرابعة: الربط Linking

الأداة المستخدمة: ld (Linker)

الوظيفة: دمج ملف Object File الخاص بك مع ملفات Object Files أخرى ضمن نفس المشروع، ومع المكتبات (مثل libc التي تحتوي على تعريف دالة printf) لإنتاج الملف النهائي.

الملف الناتج: الملف التنفيذي النهائي Executable Binary.

✅
ماذا يجب أن تتذكر؟ — أي خطأ في كتابة الكود مثل نسيان فاصلة منقوطة Syntax Error يتم اكتشافه في المرحلة الثانية Compilation. أما أخطاء إيجاد دوال (مثل إضافة دالة غير موجودة أو لم تتم إضافتها) فتظهر في المرحلة الرابعة Linking.
⚡
مراجعة سريعة: Preprocessor ➡ Compiler ➡ Assembler ➡ Linker

3 التحكم اليدوي بمراحل البناء باستخدام GCC Flags

كمس engineer أنظمة، يجب أن تعرف دائماً كيف توقف العملية في كل مرحلة للتحقق من الكود وتحليله Debug. لا يجب أن تستخدم gcc main.c فقط وتقفز للنتيجة النهائية مباشرة.

أوامر GCC للتحكم بالمراحل

Terminal
gcc -E main.c -o main.i
  • الوظيفة: إيقاف العملية بعد مرحلة Preprocessing فقط (المرحلة 1)
  • الاستخدام بشدة: يُستخدم Best Practice لاكتشاف أخطاء Macros إذا كان الماكرو الخاص بك لا يعطي النتيجة المطلوبة، استخدم -E لترى كيف قام المترجم بتوسيعه وفهمه
Terminal
gcc -S main.c -o main.s
  • الوظيفة: الترجمة حتى مرحلة Assembly والتوقف (المرحلة 2)
  • الاستخدام: لرؤية كود Assembly والتأكد من كيفية قيام المترجم بعمل Optimization أو تحسين الكود الذي كتبته
Terminal
gcc -c main.c -o main.o
  • الوظيفة: بناء Object File بدون تشغيل Linker (المرحلة 3)
  • الاستخدام: هذا الخيار بالغ الأهمية وسنستخدمه دائماً لاحقاً في أنظمة البناء Makefiles. نقوم بعمل -c لكل ملف .c بحيث لو عدلنا ملفاً واحداً نترجمه لوحده دون إعادة ترجمة المشروع كاملة
Terminal
gcc main.o utils.o -o my_app
  • الوظيفة: مرحلة Linking (المرحلة 4). يأخذ ملفات الأوبجكت ويدمجها ليصنع التطبيق التنفيذي my_app
⚠️ أخطاء شائعة

محاولة قراءة الملف التنفيذي أو ملف .o باستخدام أوامر نصية مثل cat أو nano ستظل لك رموز مشوهة لأنه ملف ثنائي Binary. يجب استخدام أدوات Binutils لقراءته.

4 الهيكل الداخلي للملفات الثنائية ELF Format & Sections

ما هو بالضبط ملف Object file أو الملف التنفيذي في لينكس؟

في وندوز، الامتداد التنفيذي هو .exe، أما في لينكس، فالمعيار المقدس للملفات التنفيذية وملفات Object يسمى ELF.

📌 تعريف

ELF (Executable and Linkable Format): هو التنسيق القياسي والهيكلي لتنظيم البيانات والكود داخل الملفات الثنائية في أنظمة لينكس.

كيف يتم تنظيم الملف من الداخل؟ The Sections ⚙

ملف ELF ليس مجرد "كتلة عشوائية" من الأصفار والآحاد، بل هو مقسم إلى أقسام أو "غرف" Sections منظمة جداً، أهمها:

.text
قسم الكود البرمجي: يحتوي على الأوامر والتعلیمات الفعلیة Instructions/Machine Code التي سینفذھا المعالج. یمتلك صلاحية Read & Execute.
.data
قسم البيانات المُهيأة: يحتوي على المتغيرات العالمية Global Variables التي تم إعطاؤها قيمة ابتدائية Initialized (مثال: int x = 5;). هذا القسم يأخذ مساحة فعلية داخل الملف في الديسك الهارد لأن القيمة يجب أن تُحفظ.
.bss
قسم Block Started by Symbol: يحتوي على المتغيرات العالمية التي لم تُعطَ قيمة ابتدائية Uninitialized (مثال: int y; التي قیمتھا صفر). ميزة عبقرية: هذا القسم لا يأخذ مساحة فعلية في الديسك الهارد! بل يحتفظ فقط بمعلومة "تخبر" الكيرنل Kernel بحجز مساحة فارغة من RAM عند تشغيل البرنامج. القسم هذا "وهمي" في الملف.
.rodata
قسم البيانات للقراءة فقط: يحتوي على الثوابت Constants مثل المتغيرات من نوع const والنصوص الحرفية المطبوعة (مثل جملة "Hello World" في دالة printf).
✅
ماذا يجب أن تتذكر؟ — المعرفة العميقة بهذه الأقسام Sections تميز المهندس المحترف في عالم الأنظمة المدمجة Embedded Systems، أحياناً نحتاج للتحكم الدقيق بوضع قسم .data في ذاكرة RAM وقسم .text في ذاكرة Flash وذلك بواسطة ما يسمى Linker Script.
⚡
مراجعة سريعة:
● .text: الكود البرمجي
● .data: متغيرات مُهيأة بقيمة
● .bss: متغيرات غير مُهيأة (لتوفير المساحة)
● .rodata: الثوابت والقراءة فقط

5 الرموز وأدوات الفحص Symbols & Binutils

المترجم يمسح كل ما هو "بشري" من الكود، ولكنه يترك أسماء الدوال والمتغيرات كعلامات تُسمى Symbols ليتمكن Linker من ربطها ببعضها لاحقاً.

أنواع الرموز Symbols

نوع الرمز الوصف
Defined Symbol (رمز معرّف) دالة أو متغير قمت بكتابة كودها الخاص داخل الملف نفسه (لها مكان في الذاكرة)
Undefined Symbol (رمز غير معرّف) دالة استخدمتھا ولكن الكود الفعلي بها غير موجود في ملفك، بل في مكتبة أو ملف أخرى. Linker هو المسؤول عن البحث عن هذا الرمز وجلبھ وتوصیلھ

أدوات الفحص والتحليل Binutils

بما أننا لا نستطيع قراءة الملفات الثنائية بالنصوص، فقد وفر لنا مشروع GNU حزمة أدوات للتحليل والفحص الهندسي:

🚀 file — تحليل نوع الملف
file my_app

الوظيفة: يحلل نوع الملف المعطى ويعطيك معلوماته (هل هو ELF؟ هل هو bit-64 أم bit-32؟ مبني لأي معمارية مثل x86_64 أم ARM؟).

🚀 nm — عرض الرموز
nm main.o

الوظيفة: يطبع قائمة بكل الرموز Symbols الموجودة في ملف Object أو الملف التنفيذي، ويوضح القسم Section الذي ينتسب إليه الرمز.

علامات هامة في المخرجات:

  • T: الرمز موجود في قسم .text (غالباً دالة مكتوبة ومعروفة)
  • U: الرمز غير معرف Undefined وينتظر Linker لجلب مشكلته من مكتبة أخرى بربطه
  • D: الرمز في قسم .data (متغير عالمي)
✅
ماذا يجب أن تتذكر؟ — أداة nm هي أسرع وأقوى وسيلة لمعرفة سبب أخطاء Linker عندما تظهر لك رسالة undefined reference. استخدمها للتأكد هل الرمز موجود أم مفقود داخل الملف أو المكتبة التي تقوم بربطه.
🚀 readelf — قراءة هيكل ELF
readelf -S main.o

الوظيفة: أداة قوية جداً تقرأ وتطبع الهيكل الداخلي لملف ELF بالكامل. خيار -S يعرض جميع الأقسام Sections مع أحجامها ومكانها الدقيق داخل الملف.

🚀 objdump — التفكيك العكسي
objdump -d main.o

الوظيفة: أداة الهندسة العكسية Disassembler تأخذ كود الآلة الثنائي Machine code من قسم text وتعيده إلى كود Assembly لتتمكن من قراءته ومعرفة كيف تمت الترجمة الفعلية.

الأداة الوظيفة مثال الاستخدام
file تحليل نوع الملف والمعمارية file my_app
nm عرض الرموز Symbols وأقسامها nm main.o
readelf -S عرض الأقسام Sections الداخلية readelf -S main.o
objdump -d التفكيك العكسي Disassembly objdump -d main.o

6 الفرق بين Native Compilation وCross-Compilation

هذا المفهوم هو الذي سيرافقك طوال مسيرتك المهنية كمهندس أنظمة مدمجة، وهو سبب وجود أداة Yocto.

كيف نعمل في بيئة Embedded Linux؟

Native Compilation — البناء المحلي
Host (جهازك — x86_64)
gcc (مترجم عادي)
Executable (x86_64)
Host = Target — نفس المعمارية ونفس الجهاز
Cross-Compilation — البناء المتقاطع
Host (جهازك — x86_64)
Cross-Compiler
Target (اللوحة — ARM)
Executable (ARM)
Host ≠ Target — معمارية مختلفة تماماً

لماذا نستخدم Cross-Compilation؟

لوحات الأنظمة المدمجة (مثل Raspberry Pi أو لوحات ST) تمتلك موارد جداً ضعيفة مقارنة بالحواسيب (معالج أضعف، ذاكرة أقل، مساحة تخزين محدودة).

إذا حاولنا تثبيت وتشغيل أداة المترجم الضخمة وعملية البناء الطويلة على اللوحة نفسها Native Compilation فقد تستغرق العملية أسابيع!

لذلك، الممارسة الأفضل هي استخدام سرعة وقوة الحواسيب الشخصية أو الخوادم لعملية البناء الثقيلة، ثم نقل الناتج النهائي (Binaries / ELF files) فقط إلى اللوحة لتشغيله.

✅
ماذا يجب أن تتذكر؟ — في Cross-compilation، جهازك القوي يسمى Host والجهاز المدمج الذي سيشغل البرنامج يسمى Target. الأداة التي نستخدمها للترجمة هي أداة مخصصة لمعمارية Target (مثال: استخدام arm-linux-gnueabihf-gcc بدلاً من gcc العادي).
وجه المقارنة Native Compilation Cross-Compilation
مكان الترجمة على الجهاز نفسه على جهاز قوي مختلف عن لوحة التطوير
المترجم gcc العادي arm-linux-gnueabihf-gcc
Host جهازك (مثلاً x86_64) جهازك (مثلاً x86_64)
Target نفس الجهاز Host = Target لوحة التطوير (مثلاً ARM) Host ≠ Target
السرعة تعتمد على موارد الجهاز سريعة جداً (تستفيد من قوة الخادم)
الاستخدام تطوير برامج سطح المكتب تطوير الأنظمة المدمجة Embedded Systems

📌 تعريفات مهمة

  • ELF (Executable and Linkable Format): التنسيق القياسي لتنظيم البيانات والكود داخل الملفات الثنائية في أنظمة لينكس

  • Target: الجهاز المدمج الذي سيشغل البرنامج النهائي
  • Cross-Compiler: أداة ترجمة مخصصة لبناء برامج لمعمارية مختلفة عن معمارية الجهاز الذي يعمل عليه

  • الخلط بين اللغات المترجمة Compiled والمفسرة Interpreted: اللغات المفسرة لا تمر بمراحل الترجمة الأربعة

💡 نصائح للحفظ
  • احفظ ترتيب المراحل بالاختصار: P-C-A-L (Preprocessor, Compiler, Assembler, Linker)
  • احفظ امتدادات الملفات بالترتيب: .c → .i → .s → .o → (executable)
  • احفظ أقسام ELF بالجملة: "Text للكود، Data للبيانات المُهيأة، BSS للفارغة (لا تأخذ مساحة)، Rodata للقراءة فقط"
  • احفظ علامات nm: T = Text (دالة)، U = Undefined (مفقود)، D = Data (متغير)
  • احفظ أوامر GCC: -E = إيقاف بعد Preprocessing، -S = إيقاف بعد Compilation، -c = إيقاف بعد Assembly
