ما هو نظام التشغيل؟
Operating System — فهم المبدأ الأساسي لأي نظام تشغيل قبل الغوص في Linux
يجب أولاً فهم المبدأ الأساسي لأي نظام تشغيل، وذلك لفهم Linux.
2.1 لماذا وُجد نظام التشغيل؟

أي حاسوب أو لوحة تطوير تحتوي على مكونات صلبة Hardware مثل CPU, RAM, Storage، لا يمكن للمستخدم أو التطبيقات التحدث مباشرة مع هذه المكونات. يأتي هنا دور نظام التشغيل OS ليكون الوسيط والمدیر للموارد.

دور نظام التشغيل كوسيط
2.2 كيف يعمل نظام التشغيل ومهامه؟ ⚙

في الأساس هو مجموعة من الأوامر. كل تطبيق Application وظيفته تنفيذ الأوامر Instructions التي يقوم المعالج CPU بتنفيذها. نظام التشغيل يقوم بالمهام التالية:

📘 المهمة 1: إدارة الذاكرة — Memory Management

النظام يوفر طريقة آمنة للقراءة والكتابة في الذاكرة.

مثال: وظيفة تخصيص مساحة في الذاكرة للمستخدم مثل دالة malloc().

📘 المهمة 2: إدارة الملفات — File System Management

ينظم كيفية حفظ وقراءة الملفات والمجلدات Files & Directories داخل وحدات التخزين Storage.

📘 المهمة 3: إدارة العمليات والمجدول — Process Management & Scheduler

تعريف Process: هو تطبيق Application في حالة تشغيل Running.

المجدول Scheduler: هو جزء ذكي من نظام التشغيل ينظم وقت المعالج لتنفيذ أوامر عدة تطبيقات في الوقت نفسه، مما يجعلك تشعر أن كل شيء يعمل بالتوازي بسرعة فائقة.

📘 المهمة 4: الشبكات والحمایة — Networking & Security

دعم بروتوكولات الاتصال وحمایة موارد النظام.

✅ تذكّر

نظام التشغيل هو المنظم والمدیر لموارد الھاردویر CPU, RAM, Storage.

📌 مراجعة سريعة ⚡
  • Process: تطبيق قيد التشغيل.
  • Scheduler: منظم وقت المعالج.
  • Memory Management: مساحات الذاكرة العشوائية مدیر.

3.2 كيف تم بناء Unix؟

في البداية كُتب بلغة التجميع Assembly.

⚠️ المشكلة

لغة Assembly مرتبطة جداً بهيكلية المعالج Architecture-specific، الكود المكتوب لمعالج معين لا يعمل على معالج آخر.

✅ الحل

قام Dennis Ritchie باختراع لغة C Programming Language وتمت إعادة كتابة نظام Unix بها.

الميزة: يمكن كتابة الكود مرة واحدة بلغة C ثم استخدام مترجم Compiler لتحويله إلى Assembly الخاص بأي معالج آخر. (هذه العملية تسمى Compilation Process).

عملية التجميع — Compilation Process
3.3 مشكلة تفتت Unix

بسبب نجاح Unix الكبير، بدأت الشركات بتطوير أنظمة مقلدة له، مما أدى إلى ظهور نسخ مختلفة كثيرة قليلاً، والتي هددت بضياع الميزة الكبيرة وهي "قابلية النقل" Portability وهي أساسية يحتاجها المطورون.

✅ تذكّر

لغة C تم ابتكارها بالأساس لحل مشكلة كتابة أنظمة التشغيل بلغة Assembly لتسهيل نقل النظام بين المعالجات المختلفة.

⚠️ أخطاء شائعة

الاعتقاد بأن Linux هو Unix نفسه (هذا خطأ فادح، سنفهم سببه لاحقاً).

CHAPTER 04
معيار POSIX وقابلية النقل
Portable Operating System Interface — القانون الذي وحّد واجهات الأنظمة وضمن قابلية النقل البرمجي
معيار POSIX هو المستند الذي يضمن أن الكود البرمجي سيعمل على أي نظام تشغيل يحترم هذا المعيار.
4.1 لماذا وُجد POSIX؟

لحل مشكلة اختلاف الأنظمة المقلدة لـ Unix، احتاج المطورون إلى "قانون" أو "تصميم هندسي" يوحد طريقة عمل هذه الأنظمة لضمان أن التطبيق المكتوب لنظام معين سيعمل على جميع الأنظمة الأخرى.

4.2 ما هو POSIX؟ ⚙

ببساطة هو عن عبارة مستند نصي (قانون/معيار) يحتوي على فصول Chapters تشرح كيف يجب أن يتصرف ويتواصل نظام التشغيل مع التطبيقات.

📘 أهم فصول معيار POSIX

1. واجهات النظام — System Interfaces: مجموعة الدوال Functions التي يجب أن يوفرها النظام للتطبيقات لتتحدث معه (مثل دوال القراءة والكتابة).

2. ملفات الترويسة — Headers: أسماء الملفات التي تحتوي على هذه الدوال ويحددها POSIX.

3. الأدوات — Utilities: التطبيقات الأساسية التي يجب أن تتواجد في واجهة سطر الأوامر Shell.

🚀 مثال عملي

بفضل POSIX، دالة الطباعة printf() موجودة دائماً داخل ملف <stdio.h> في أي نظام تشغيل يحترم هذا المعيار.

الأوامر الأساسية Linux — Utilities المذكورة كمثال
الأمر الوظیفة الاستخدام
ls عرض محتوى المجلد List directory contents هذا الأمر يستعمل دوال من واجهة النظام للوصول إلى الديسك الھارد لجلب أسماء الملفات وعرضها
cd تغییر المجلد الحالي Change Directory للانتقال من مسار إلى آخر
mkdir إنشاء مجلد جدید Make Directory —
بنية POSIX — كيف يوحّد المعيار الأنظمة
4.3 ميزة قابلية النقل — "Portability"

إذا كتبت كوداً بلغة C يستخدم دالة printf() و #include <stdio.h>، أصبح الكود قابلاً للنقل Portable بفضل POSIX. فأنت ضامن %100 أن هذا الكود سيعمل على أي نظام تشغيل مبني على معيار POSIX.

✅ تذكّر

POSIX هو مجرد مستند قانون لتوحيد واجهات الأنظمة لضمان قابلية النقل Portability للكود البرمجي، وليس نظام تشغيل.

CHAPTER 05
مشروع GNU ورخصة GPL
Free Software Foundation — كيف وُلدت فلسفة البرمجيات الحرة وأدوات GNU التي تشكل العمود الفقري لنظام Linux
من أزمة "الثقة" في الأنظمة المغلقة إلى بناء بديل مفتوح المصدر بالكامل ومتوافق مع معيار POSIX.
5.1 المشكلة مع Unix

نظام Unix لم يكن مفتوح المصدر Closed Source، كمطور أو مستخدم لا يمكنك رؤية الكود البرمجي الخاص به، مما يخلق أزمة "ثقة" (مثلما يفعل نظام Windows) لا تستطيع التأكد %100.

5.2 ظهور Richard Stallman ومشروع GNU

Richard Stallman رفض مبدأ البرمجيات المغلقة وأسّس فلسفة البرمجيات الحرة Free Software.

أطلق مشروعاً ضخماً اسمه GNU (وهي اختصار لـ GNU's Not Unix — أي ليس يونكس).

الهدف: بناء نظام تشغيل متوافق مع معيار POSIX لكنه مفتوح المصدر بالكامل.

5.3 ماذا طوّر مشروع GNU؟

بما أن POSIX مجرد "قانون" نصي، فكان على Stallman كتابة كود برمجي يطبق هذا القانون Implementation.

📘 1. مكتبة C — C Library (glibc)

تطبيق فعلي Implementation لواجهات النظام والترويسة وملفات المذكورة في POSIX (وتعد glibc مكتبة مقدسة في عالم Linux).

📘 2. Coreutils

تطبيق للأدوات Utilities مثل ls، cd، mkdir.

📘 3. المترجم GCC

كان في البداية GNU C Compiler ثم تطور ليصبح GNU Compiler Collection وأشهر وأقوى مترجم في التاريخ.

5.4 رخصة GPL — GNU General Public License

لماذا وُجدت؟ لحماية حق العمیل في الحصول على الكود المصدري المفتوح بقوة القانون.

كيف تعمل؟ أي برنامج يُنشر تحت رخصة GPL يجب أن يكون كوده المصدري متاحاً لأي شخص يستخدمه، وإذا قام شخص بتعديله وبناء برنامج جديد عليه، يجب أن يكون البرنامج الجديد مفتوح المصدر أيضاً.

✅ تذكّر

مكتبة glibc هي مجرد Implementation واحد لمعيار POSIX، يوجد هناك مكتبات أخرى تطبق نفس المعيار مثل musl أو micro libc.

CHAPTER 06
نواة Linux وتكامل GNU/Linux
كيف اكتملت الصورة — من مشروع GNU الناقص إلى ولادة نظام التشغيل الأقوى في العالم
عندما اجتمعت النواة Linux مع تطبيقات ومكتبات GNU وُلد نظام التشغيل الكامل الذي نستخدمه اليوم.
6.1 القطعة الناقصة في مشروع GNU

مشروع GNU كان لديه كل شيء (المكتبات، الأدوات، المترجم) باستثناء الجزء الأهم: نواة نظام التشغيل Kernel (محاولتهم باءت بالفشل لبناء نواة اسمها Hurd).

6.2 ظهور Linux (Linus Torvalds 1991)

قام الطالب الفنلندي Linus Torvalds بتطوير نواة نظام تشغيل Kernel واستخدم مترجم GCC لتطويرها ورخصها ورخصها وجعلها متوافقة مع POSIX. أطلق عليها اسم Linux Kernel.

بنية GNU/Linux — كيف تكتمل الصورة

عندما اجتمعت النواة Linux مع تطبيقات ومكتبات GNU وُلد نظام التشغيل الكامل الذي نستخدمه اليوم. السبب الذي يجعل Stallman يدافع بشراسة على تسمية النظام GNU/Linux وليس Linux فقط، لأن الجهد الأكبر من ناحیة الأدوات والمترجمات جاء من GNU.

6.3 مقارنات تقنية
مقارنة: Unix-Compliant مقابل Unix-like
المعيار النظام المطابق لـ Unix النظام الشبيه بـ Unix
تطبيق المعيار يطبق معيار POSIX بحذافيره (%100) يشبه نظام Unix ويطبق معيار POSIX بنسبة عالية (%95-98) لكن ليس كلياً
الاعتماد مر بمراحل اختبار Testing واجتاز شھادة الاعتماد Certification الرسمية لا يمتلك الشھادة لأنه لا يملك الحق في استخدام اسم Unix (Trade mark)
الميزات الإضافية لا يضيف به خاصة ميزات غير موجودة في POSIX يضيف ميزات خاصة به
مثال macOS Linux
مقارنة: Kernel Space مقابل User Space
الخاصية مساحة النواة — Kernel Space مساحة المستخدم — User Space
التعریف المكان الذي تعیش وتعمل فیه النواة Kernel المكان الذي تعمل فیه تطبیقات المستخدم العادیة
الصلاحیات صلاحیات كاملة ومطلقة للتحكم في العتاد Hardware لا یملك أي صلاحیة للتحكم أو للتحول مباشرة بالعتاد
طریقة العمل ینفذ الحساسة الأوامر (إدارة ذاكرة، بروتوكولات، الخ) إذا احتاج التطبيق لشيء من العتاد، يجب أن یطلب ذلك من Kernel عبر النظام واجھات
مساحة النواة مقابل مساحة المستخدم
🧠 فهم عميق

هل يمكن في العصر الحالي أن نقول "Linux" بدون "GNU"؟ نعم، خاصة في Embedded Linux يمكنك بناء نظام يتكون من:

  • Kernel: Linux Kernel
  • Compiler: Clang (بدلاً من GCC)
  • C Library: musl (بدلاً من glibc)
  • Utilities: BusyBox (بدلاً من Coreutils) — وهو أسرع وأخف جداً ومناسب للأنظمة المدمجة

في هذه الحالة النظام لا يستخدم أجزاء GNU ويسمى Linux فقط.

✅ تذكّر

تقنياً كلمة Linux تعني النواة Kernel فقط. نظام التشغيل الكامل يتكون من النواة مضافاً إلیھا مكتبات وأدوات ومساحة مستخدم User Space.

⚠️ أخطاء شائعة

الخلط بين gcc (كـ Executable compiler) و GCC (كـ Compiler Collection) — الكبیرة والأداة هي المشروع بأكمله، ولیست الصغیرة الكلمة.

❓ لماذا وُجد نظام التشغيل؟

أي حاسوب أو لوحة تطوير تحتوي على مكونات صلبة Hardware مثل CPU, RAM, Storage، لا يمكن للمستخدم أو التطبيقات التحدث مباشرة مع هذه المكونات. يأتي هنا دور نظام التشغيل OS ليكون الوسيط والمدیر لهذه الموارد.

⚙️ كيف يعمل نظام التشغيل وما مهامه؟

في الأساس كل تطبيق Application وظیفته تنفیذ الأوامر Instructions، المعالج CPU يقوم بنظام من مجموعة الأوامر. نظام التشغيل يقوم بالمهام التالية:

1. إدارة الذاكرة Memory Management

النظام یوفر طریقة آمنة للقراءة والكتابة في الذاكرة.

🚀
مثال عملي
وظیفة تخصیص مساحة في الذاكرة للمستخدم مثل دالة malloc()

2. إدارة الملفات File System Management

ینظم كیفیة حفظ وقراءة الملفات والمجلدات Files & Directories داخل وحدات التخزین Storage.

3. إدارة العملیات والمجدول Process Management & Scheduler

📌
تعريف
Process: تطبیق Application هو في حالة تشغیل Running.

المجدول Scheduler: هو جزء ذكي من نظام التشغیل ینظم وقت المعالج لتنفیذ أوامر عدة تطبیقات في نفس الوقت، مما یجعلك تشعر أن كل شيء یعمل بالتوازي بسرعة فائقة.

4. الشبكات والحمایة Networking & Security

دعم بروتوكولات الاتصال وحمایة موارد النظام.

✅
تتذكر؟ ماذا يجب أن
نظام التشغیل هو المنظم والمدیر لموارد الھاردویر CPU, RAM, Storage.
📘
مراجعة سريعة
  • Process: تطبیق قید التشغیل.
  • Scheduler: منظم وقت المعالج.
  • Memory Management: مساحات الذاكرة العشوائیة مدیر.

❓ لماذا وُجد POSIX؟

لحل مشكلة اختلاف الأنظمة المقلدة لـ Unix، احتاج المطورون إلى "قانون" أو "تصمیم ھندسي" یوحد طریقة عمل ھذه الأنظمة لضمان أن التطبیق المكتوب لنظام معین سیعمل على جمیع الأنظمة الأخرى.

⚙️ ما هو POSIX؟

ببساطة ھو عبارة عن مستند نصي (قانون/معیار) یحتوي على فصول Chapters تشرح كیف یجب أن یتصرف ویتواصل نظام التشغیل مع التطبیقات.

🧠
فهم عميق
POSIX هو مستند مجرد وليس نظام تشغیل ولا كود برمجي، مجرد قانون لضمان قابلية النقل Portability للكود البرمجي.

📖 أهم فصول معيار POSIX

1. واجھات النظام System Interfaces

مجموعة من الدوال Functions یجب أن یوفرھا النظام للتطبیقات لتتحدث معھ (مثل دوال القراءة والكتابة).

2. ملفات الترویسة Headers

أسماء الملفات التي تحتوي على الدوال ھذه. POSIX یحدد.

🚀
مثال عملي
بفضل POSIX، دالة الطباعة printf موجودة دائماً داخل ملف <stdio.h> في أي نظام تشغیل یحترم ھذا المعیار.

3. الأدوات Utilities

التطبیقات الأساسیة التي یجب أن تتواجد في واجهة الأوامر سطر Shell.

الأوامر الأساسیة في Linux (التی ذكرت كأدوات Utilities):

الأمر الوظیفة بالإنجليزية الوصف الاستخدام
ls List directory contents عرض محتوى المجلد یستعمل الأمر ھذا دوال من واجھة النظام للوصول إلى الھارد دیسك وجلب أسماء الملفات وعرضھا
cd Change Directory تغییر المجلد الحالي للانتقال من مسار إلى آخر
mkdir Make Directory إنشاء مجلد جدید —

🚀 ميزة قابلية النقل Portability

إذا كتبت كوداً بلغة C یستخدم دالة printf و #include <stdio.h>، أصبح الكود قابلاً للنقل Portable بفضل POSIX. فأنت ضامن %100 أن ھذا الكود سیعمل على أي نظام تشغیل مبني على معیار POSIX.

C 
#include <stdio.h>

int main() {
    printf("Hello, Embedded Linux!\n");
    return 0;
}
✅
تتذكر؟ ماذا يجب أن
POSIX ھو مجرد مستند قانون لضمان قابلية النقل للكود البرمجي، وليس نظام تشغیل ولا كود برمجي.

🔒 المشكلة مع Unix

نظام Unix لم یكن مفتوح المصدر Closed Source، لا يمكنك رؤیة الكود البرمجي الخاص بھ كمطور أو مستخدم، لا تستطیع التأكد %100 (مثلما یفعلھ نظام Windows في الخلفیة) مما یخلق أزمة "ثقة".

👤 ظهور Richard Stallman ومشروع GNU

Richard Stallman رفض مبدأ البرمجیات المغلقة وأسس فلسفة البرمجیات الحرة Free Software.

أطلق مشروعاً ضخماً اسمه GNU (وهي اختصار لـ GNU's Not Unix — أي أن GNU لیس یونكس).

📘
الهدف
بناء نظام تشغیل متوافق مع معیار POSIX لكنه مفتوح المصدر بالكامل.

🛠️ ماذا طوّر مشروع GNU؟

بما أن POSIX مجرد "قانون" نصي، كان على Stallman كتابة كود برمجي یطبق ھذا القانون Implementation.

1. مكتبة C Library (glibc)

تطبیق فعلي Implementation لواجھات النظام والترویسة وملفات المذكورة في POSIX (وتعد glibc مكتبة مقدسة في عالم Linux).

2. Coreutils

تطبیق للأدوات Utilities مثل ls، cd، mkdir.

3. المترجم GCC

كان في البدایة GNU C Compiler ثم تطور لیصبح GNU Compiler Collection وأشھر وأقوى مترجم في التاریخ.

📜 رخصة GPL (GNU General Public License)

لماذا وُجدت؟ لحمایة حق العمیل في الحصول على الكود المصدري المفتوح بقوة القانون.

كیف تعمل؟ أي برنامج یُنشر تحت رخصة GPL یجب أن یكون كوده المصدري متاحاً لأي شخص یستخدمھ، وإذا قام شخص بتعدیلھ وبناء برنامج جدید علیھ، یجب أن یكون البرنامج الجدید أیضاً مفتوح المصدر.

✅
تتذكر؟ ماذا يجب أن
مكتبة glibc هی مجرد Implementation واحد لمعیار POSIX، یوجد ھناك مكتبات أخرى تطبق نفس المعیار مثل musl libc أو micro libc.

❓ القطعة الناقصة في مشروع GNU

مشروع GNU لدیه كل شيء إلا أھم جزء: نواة نظام التشغیل Kernel (المكتبات، الأدوات، المترجم، الرخصة)، باستثناء محاولتھم لبناء نواة اسمھا Hurd التي باءت بالفشل.

🌟 ظهور Linus Torvalds (1991)

قام الطالب الفنلندي Linus Torvalds بتطویر نواة نظام تشغیل Kernel واستخدم مترجم GCC لتطویرھا ورخصھا تحت GPL وجعلھا متوافقة مع POSIX. أطلق علیھا اسم Linux Kernel.

عندما اجتمعت النواة Linux مع تطبیقات ومكتبات GNU، وُلد النظام التشغیل الكامل الذي نستخدمھ الیوم.

🧠
فهم عميق
السبب لھذا أن Stallman یدافع بشراسة على تسمیة النظام GNU/Linux ولیس Linux فقط، لأن الجھد الأكبر من ناحیة الأدوات والمترجمات والمكتبات جاء من GNU.

مقارنة: Unix-Compliant مقابل Unix-like

المعيار Unix-Compliant — لیونكس المطابق Unix-like — بیونكس الشبیھ
التعريف نظام یطبق معیار POSIX بحذافیره (%100 عالیة بنسبة) نظام یشبھ Unix ویطبق معیار POSIX لكنه لیس كلیاً (%95-98)
الاعتماد مر بمراحل اختبار Testing واجتاز شھادة التوافق لا یمتلك شھادة رسمیة
المیزات لا یضیف میزات خاصة بھ غیر موجودة في POSIX یضیف میزات خاصة بھ غیر موجودة في POSIX
استخدام اسم Unix یملك الحق في استخدام اسم Unix (Trade mark) لا یمتلك الحق لأنه لا یمتلك الشھادة
مثال macOS Linux

مقارنة: Kernel Space مقابل User Space

المعيار Kernel Space — النواة مساحة User Space — المستخدم مساحة
التعريف المكان الذي تعیش وتعمل فیھ النواة Kernel المكان الذي تعمل فیھ تطبیقات المستخدم العادیة
الصلاحیات صلاحیات كاملة ومطلقة للتحكم في العتاد Hardware لا یملك أي صلاحیة للتحكم أو للتحول مباشرة بالعتاد
طریقة العمل ینفذ الحساسة الأوامر (إدارة ذاكرة، بروتوكولات، الخ) إذا احتاج التطبیق لشيء من العتاد، یجب أن یطلب ذلك من Kernel الـ عبر النظام واجھات

❓ هل يمكن أن نقول "GNU" بدون "Linux" في العصر الحالي؟

نعم، خاصة في Embedded Linux الـ يمكنك بناء نظام من:

  • Kernel: Linux Kernel
  • Compiler: Clang (بدلاً من GCC)
  • C Library: musl (بدلاً من glibc)
  • Utilities: BusyBox (بدلاً من Coreutils) وهو أسرع وأخف جداً ومناسب للأنظمة المدمجة

في الحالة ھذه النظام لا یستخدم أجزاء GNU ویسمى Linux فقط.

✅
تتذكر؟ ماذا يجب أن
كلمة Linux تقنیاً تعني النواة Kernel فقط. النظام التشغیل الكامل یتكون من النواة plus إلیھا مضافا مكتبات وأدوات مساحة المستخدم User Space.
⚠️
أخطاء شائعة
الخلط بین gcc (كـ Executable compiler) و GCC (كـ Compiler Collection) — الأداة الكبیرة والصغیرة الكلمة ھي المشروع بأكملھ.

إجابات على أكثر الأسئلة التي تطرح عند البدء في مجال Embedded Linux

❓ الأسئلة والأجوبة

📘
سؤال: ما الفرق بين رخصة MIT ورخصة GPL؟
رخصة MIT تمتلك قیوداً أقل Less limitations وتعطیك حریة أكبر في التصرف في الكود المصدري مقارنة بـ GPL التي تجبرك على إبقاء الكود مفتوحاً في المشاریع المشتقة.
📘
سؤال: هل نظام Windows يملك Kernel؟
نعم، أي نظام تشغیل في العالم (Windows, macOS, Linux) یمتلك نواة Kernel مھمتھا التحكم بالعتاد.
📘
سؤال: ما هي متطلبات أداة Yocto؟
حاسوباً قویاً جداً؛ الحد الأدنى RAM 8GB (ویفضل جداً 16GB) ومعالج لا یقل عن 4 أنویة، لأن كلما كانت عملیة البناء أقوى كان أسرع.
📘
سؤال: هل يمكن الاعتماد على المحاكاة Simulation بدون لوحة حقيقية؟
سنبدأ في Yocto باستخدام محاكي اسمه QEMU، ولكن في مراحل كتابة الـ Drivers والتعامل مع السنسورات (I2C, SPI)، یجب أن تمتلك عتاداً حقیقیاً.
📘
سؤال: ما الفرق بين أنظمة مثل Zephyr و Embedded Linux؟
Zephyr موجھ للأنظمة الدقیقة جداً RTOS لأن المبادئ Concepts مختلفة، ولكن ھناك بعض التشابھات مثل Device Tree و Compilation process.

📝 مراجعة سريعة للدرس
📌 المفاهيم الرئيسية
  • نظام التشغیل هو الوسیط والمدیر بین التطبیقات والمكونات الصلبة Hardware.
  • لغة C ابتُكرت لحل مشكلة نقل أنظمة التشغیل بین المعالجات المختلفة بدلاً من Assembly.
  • POSIX هو معیار نصي یوحد واجھات الأنظمة لضمان قابلية نقل الكود Portability.
  • مشروع GNU وفّر المكتبات glibc والأدوات Coreutils والمترجم GCC لكنه افتقر إلى النواة.
  • Linux Kernel كملت القطعة الناقصة لتشكیل نظام GNU/Linux الكامل.
  • Kernel Space یمتلك صلاحیات مطلقة، User Space یحتاج System Calls للوصول إلى العتاد.
  • في Embedded Linux يمكن استخدام بدائل GNU مثل musl و BusyBox و Clang.
📌 التعريفات المهمة
  • Process: تطبیق Application في حالة تشغیل Running.
  • Scheduler: جزء ذكي من نظام التشغیل ینظم وقت المعالج بین العملیات.
  • Memory Management: تنظیم القراءة والكتابة في الذاكرة العشوائیة بشكل آمن.
  • POSIX: مستند نصي (قانون) یحدد واجھات النظام والترویسة ملفات والأدوات الأساسیة.
  • GNU: مشروع لبناء نظام تشغیل حر ومتوافق مع POSIX (GNU's Not Unix).
  • GPL: رخصة تضمن بقاء الكود المفتوح حتى في المشاریع المشتقة.
  • Kernel: الجزء الأساسي من نظام التشغیل الذي یتحكم مباشرة بالعتاد.
  • Kernel Space: مساحة النواة بصلاحیات مطلقة.
  • User Space: مساحة التطبیقات العادیة بصلاحیات محدودة.
  • Portability: قدرة الكود على العمل على أنظمة ومعالجات مختلفة.
  • Compilation Process: تحویل الكود المصدري Source Code إلى ملف تنفيذي Executable عبر المترجم والرابط.
  • GCC: GNU Compiler Collection — مجموعة مترجمات (لیس مجرد مترجم C واحد).
⚠️ أخطاء شائعة
شراء إصدار قدیم من Raspberry Pi (مثل النسخة الأولى Zero 1) بدلاً من Zero 2 W — یسبب مشاكل في الأداء والتوافق.
الاعتقاد بأن Linux هو Unix نفسه — Linux نظام شبیھ بـ Unix (Unix-like) ولیس مطابقاً له.
الخلط بین gcc (المترجم التنفيذي الصغیر) و GCC (مشروع المجموعة الكبیرة).
اعتبار أن POSIX هو نظام تشغیل أو كود — هو مجرد مستند معیاري نصي.
🎯 ملاحظات للمقابلات
لماذا تُعد لغة C أساسیة في أنظمة التشغیل؟ — لأنها توفر تحكماً منخفض المستوى بالعتاد مع قابلية النقل Portability بین المعالجات.
ما الفرق بین Kernel Space و User Space؟ — Kernel Space یعمل بصراحة كاملة مباشرة مع العتاد، بینما User Space یحتاج استدعاءات نظام System Calls.
هل Linux هو Unix؟ — لا، Linux هو Unix-like، لا یمتلك شھادة Unix الرسمیة.
لماذا یدافع Stallman على تسمیة GNU/Linux؟ — لأن أغلب مكونات النظام (المكتبات، الأدوات، المترجم) من مشروع GNU.
ما هو POSIX ولماذا ھو مهم في Embedded Linux؟ — یضمن أن الكود المكتوب لنظام ینتقل إلى آخر بدون تعدیل.
💡 نصائح للحفظ
GNU = GNU's Not Unix — الاسم نفسه یذكرك أنه مشروع مختلف عن Unix لكنه متوافق معه.
POSIX = مستند قانون مثل "الدستور" — الأنظمة تتبعه لكنه لیس نظاماً بحد ذاته.
Kernel = "النواة" — قلب النظام الذي یتحكم بالعتاد مباشرة.
gcc صغیر = مترجم واحد، GCC كبیر = مجموعة كاملة من المترجمات.
تذكر التسلسل: Multics → Unix → C Language → POSIX → GNU → Linux → GNU/Linux.
📊 نصائح للامتحانات
تركّز الامتحانات على الفرق بین Unix-Compliant و Unix-like — تذكر أن macOS مطابق و Linux شبیھ.
مھام نظام التشغیل الأربعة: إدارة الذاكرة، إدارة الملفات، إدارة العملیات والجدولة، الشبكات والأمان.
components مشروع GNU الثلاثة: glibc + Coreutils + GCC.
في Embedded Linux قد نستبدل: glibc → musl، Coreutils → BusyBox، GCC → Clang.
