1 التواصل مع برامج تعريف النواة: ioctl & Kernel Headers

قبل الدخول في برمجة الـ Shell، أجاب المدرب على سؤال جوهري حول كيفية تفاعل التطبيقات مع العتاد عبر الكيرنل: كيف تتواصل التطبيقات User Space مع الـ Kernel Drivers؟؟

سابقاً تعلمنا أننا نستخدم نداءات النظام System Calls مثل read و write. ولكن ماذا لو كنا نريد إرسال أمر "خاص" لقطعة هاردوير (مثال: تغيير سرعة دوران محرك، أو ضبط إعدادات حساس) لا يمكن التعبير عنه بمجرد كتابة أو قراءة؟

📌 تعريف: نداء النظام ioctl — Input/Output Control

هو نداء نظام خاص Special System Call يُستخدم لإرسال أوامر مخصصة Custom Commands لبرامج التعريف Drivers داخل الكيرنل.

  • ioctl يعتمد على معرفات IDs — يقوم التطبيق بإرسال ID معين، ويقوم الـ Driver بالتقاط هذا الـ ID ومطابقة تماماً لما ينتظره الـ Driver في الـ Kernel Space (حسب معيار ABI).
  • ثم ينفذ الوظيفة المقابلة لهذا الـ ID.

ما هي Kernel Headers؟ ولماذا توجد؟

📌 تعريف: Kernel Headers

هي ملفات ترويسة Header Files .h مكتوبة بلغة C، يتم استخراجها من الكيرنل وإتاحتها لمساحة المستخدم User Space.

وظيفتها:

  • تحتوي على تعريفات الثوابت Constants وأرقام الـ IDs الخاصة بالـ ioctl.
  • ليتمكن المطور من عمل include لها في تطبيقه، مما يضمن توافق الأوامر بين التطبيق والكيرنل.

نداء ioctl هو نافذة التطبيقات للتحكم الدقيق والمخصص في العتاد عبر الـ Drivers. والـ Kernel Headers هي "القاموس" المشترك بين التطبيق والكيرنل.

💡 مراجعة سريعة
  • read/write: لنقل البيانات القياسية.
  • ioctl: لإرسال أوامر مخصصة وإعدادات للعتاد.
  • Kernel Headers: ملفات .h تضمن اتفاق التطبيق والكيرنل على نفس الثوابت.

2 الجملة الشرطية if Statement في الشل

📘 مفهوم أساسي

الجملة الشرطية في الـ Shell تختلف جذرياً فلسفياً عن لغات البرمجة العادية مثل C أو Python.

كيف ينظر الشل إلى الجملة الشرطية if؟

في لغة C

if تقوم بتقييم متغير أو عملية حسابية — إذا كانت النتيجة 0 ف هي False، وإلا ف هي True.

VS

في الـ Shell

الشرط الخاص بـ if عبارة عن "أمر" Command.

ماذا يحدث بالضبط؟

  • الـ Shell يقوم بتشغيل الأمر.
  • ينظر إلى حالة الخروج Exit Status (الموجودة في المتغیر $?).
  • إذا كانت حالة الخروج تساوي 0 (نفذ الأمر بنجاح تام) — تُعتبر النتيجة True ويدخل إلى بلوك الـ then.
  • إذا كانت حالة الخروج أي رقم آخر (حدث خطأ) — تُعتبر النتيجة False ويدخل إلى بلوك الـ else.

الهيكل العام Syntax

Shell
if [Command]; then # Commands to execute if True (Exit status is 0) elif [Command]; then # Commands to execute if True else # Commands to execute if all above are False (Exit status != 0) fi
  • الوظيفة: التحكم بمسار تنفيذ البرنامج بناءً على نجاح أو فشل أوامر أخرى.
  • if: بداية الشرط (تُتبع بأمر).
  • then: بداية البلوك الذي سيُنفذ في حال النجاح.
  • fi: نهاية وإعلان إغلاق الجملة الشرطية بالكامل — وتُستخدم كلمة if معكوسة.

قاعدة السطر الواحد Semicolon Rule

⚠️ تنبيه مهم

بما أن مترجم الـ Shell Interpreter فإنه يقرأ سطراً بسطر. إذا أردت كتابة if و then في نفس السطر، يجب وضع فاصلة منقوطة ; بعد الأمر لإنهاء الجملة، ليتمكن الشل من تمييز كلمة then ككلمة مفتاحية Keyword وليس كمدخل Argument للأمر.

🚀 مثال عملي
if echo "Hi"; then echo "Done"; fi
✅ ماذا يجب أن تتذكر؟

الـ if في الـ Shell لا تقارن أرقاماً بل تشغل أمراً وتراقب نجاحه أو فشله.

⚠️ أخطاء شائعة

كتابة if VAR == 1 بناءً خاطئ تماماً لأن الـ if تنتظر "أمراً" ولا تفهم المقارنات الرياضية المباشرة.

3 أمر الفحص والمقارنة test و [ ]

بما أن الـ if تتطلب "أمراً" للعمل، فكيف يمكننا إجراء مقارنات نصوص أو أرقام؟ الحل هو استخدام أمر يسمى test.

ما هو أمر test؟

📌 تعريف: أمر test

أداة للمقارنة (للأرقام والنصوص) وفحص حالة الملفات (هل الملف موجود؟ هل هو مجلد؟) وإرجاع نتيجة Exit status.

Shell
test [Options] [Arguments]

الوظيفة: إذا كانت المقارنة صحيحة يُرجع 0، وإذا كانت خاطئة يُرجع 1.

خيارات أمر test — Options

جدول خيارات أمر test الكامل
نوع الفحص الخيار Option المعنى والوظيفة مثال
النصوص Strings -z Zero — هل النص فارغ؟ (طوله صفر) test -z "$NAME"
النصوص Strings -n Non-zero/Not empty — هل النص يحتوي على قيمة؟ test -n "$NAME"
النصوص Strings = هل النص الأول يطابق النص الثاني؟ test "$A" = "$B"
الأرقام Integers -eq Equal — هل الرقمان متساويان؟ test 1 -eq 1
الأرقام Integers -lt Less Than — هل أصغر من؟ test 10 -lt 20
الأرقام Integers -gt Greater Than — هل أكبر من؟ test 20 -gt 10
الملفات Files -d هل المسار موجود وهو مجلد Directory؟ test -d /home
الملفات Files -f هل المسار موجود وهو ملف عادي Regular File؟ test -f /etc/passwd

القوس المربع [ ] — The Bracket Command

وجد المبرمجون أن كتابة if test -n "$VAR" تبدو غير مقروءة بوضوح. تم تطوير أمر جديد تماماً مطابق لأمر test يسمى [ (نعم، اسم الأمر هو القوس المفتوح [ ولكنّه).

Shell
[ $A -eq $B ]
⚠️ قاعدة المسافات الصارمة

أمر [ يتطلب وجود مسافة Space فوراً بعده، ومسافة قبل القوس المغلق ].

السبب التقني: لأن [ هو اسم البرنامج (الكوماند)، والمدخلات هي المقارنة و Arguments، و ] هو المدخل الإجباري الأخير الذي يطلبه البرنامج لرؤيته حتى يعرف نهاية العملية. بدون مسافات الشل سيعتبرها كلمة واحدة ولن يجد أمراً بهذا الاسم.

✅ ماذا يجب أن تتذكر؟

القوس [ ليس جزءاً من قواعد الشل Syntax بل هو برنامج تنفيذي Command وظيفته المقارنة، والمسافات حوله إجبارية.

القاعدة الذهبية: علامات الاقتباس المزدوجة Double Quotes

⚠️ هذا الخطأ يقع فيه 99% من المبتدئين

يجب دائماً إحاطة المتغيرات بعلامات اقتباس مزدوجة عند استخدامها مع أمر test أو [ ].

مثال خاطئ:

[ -z $NAME ]

المشكلة: إذا كان المتغير NAME فارغاً، الشل سيُعوّضه بفراغ، وبعد الترجمة سيصبح الأمر هكذا: [ -z ] — الآن الأمر ناقص مدخل بعد -z (يحتاج ليفحصه)، مما سيؤدي إلى خطأ تشغيلي Syntax error وإرجاع نتيجة خاطئة دائماً.

الحل الصحيح:

[ -z "$NAME" ]

لو كان المتغير فارغاً، الأمر سيصبح [ -z "" ] — الآن الأمر مكتمل برمجياً وسيفحص نصاً فارغاً وسيرجع النتيجة المتوقعة بشكل سليم.

⚠️ أخطاء شائعة
  • كتابة [$A -eq $B] (بدون مسافات).
  • نسيان وضع $VAR داخل علامات اقتباس مزدوجة.

4 حلقات التكرار Loops والرموز الخاصة

حلقة for

تُستخدم لتكرار تنفيذ بلوك من الأوامر على قائمة List من العناصر.

Shell
for COUNTER in LIST; do # Commands to execute using $COUNTER done
  • الوظيفة: الشل يمر على عناصر القائمة LIST واحداً تلو الآخر، وفي كل مرة يضع العنصر في المتغير COUNTER وينفذ الأوامر الموجودة بين do و done.

طرق تكوين القائمة List Creation

  • كتابة يدوية:
    Shell
    for name in Ali Talal; do echo "$name"; done
  • استخدام أمر seq: يولد تسلسلاً رقمیاً. يتم دمجه بأقواس $() لتنفيذ الأوامر.
    Shell
    for num in $(seq 1 100); do echo "$num"; done
  • استخدام أقواس التسلسل {}:
    Shell
    for num in {1..100}; do echo "$num"; done

    • الرمز الخاص * — Wildcard

    📘 مفهوم

    رمز النجمة * في الشل يعني "كل شيء" Matches everything.

    🚀 مثال عملي
    for file in *.c; do echo "$file"; done

    التفسير: الشل سيبحث في المجلد الحالي عن جميع الملفات التي تنتهي بـ .c ويحولها إلى قائمة ليمر عليها أمر for — مفيدة جداً لمعالجة الملفات دفعة واحدة (مثل مسح جميع ملفات نوع معين rm *.txt).

    حلقة while

    حلقة التكرار المشروطة — تدمج فكرة if و for معاً.

    Shell
    while [Command]; do # Commands to execute as long as Command exits with 0 (True) done
    • الوظيفة: تستمر في تنفيذ البلوك طالما أن "الأمر" في الشرط يرجع حالة خروج 0 Success/True.

    أمران: true و false

    هما برنامجان صغيران جداً في لينكس، وظيفتها الوحيدة هي إرجاع حالة خروج محددة:

    true

    أمر يُرجع دائماً 0 (نجاح).

    VS

    false

    أمر يُرجع دائماً 1 (فشل).

    الاستخدام: لإنشاء حلقة تكرار لا نهائية Infinite Loop.

    Shell
    while true; do echo "Running forever..." done

    5 العمليات الحسابية ومعياريّة POSIX وخطر استخدام Bashisms

    بما أن الشل يعتبر كل شيء نصوصاً Strings، فكيف نطرح أو نجمع أرقاماً في الشل؟

    الطرق المتاحة

  • القوسان المزدوجان $(( )): تُستخدم لإجراء العمليات الحسابية مباشرة.
    Shell
    echo $(( 1 + 1 ))
  • أمر expr: برنامج يقوم بتقييم العمليات الحسابية.
    Shell
    echo $(expr 1 + 1)

    • التحذير الأهم للمهندس: POSIX Compliance vs Bashisms

    📘 مفهوم أساسي

    برنامج Bash Bourne Again Shell هو الـ Shell الافتراضي في العديد من التوزيعات (مثل Ubuntu). مطورو Bash قاموا بإضافة العديد من المزايا التي تسهل الكتابة، ولكنها ليست جزءاً من المعيار القياسي POSIX وتسمى Bashisms.

    🚀 مثال على Bashism
    # حلقات تكرار تشبه C: for ((i=0; i<10; i++)) # استخدام المقارنة المزدوجة [[ ]]: [[ $VAR == "hello" ]]
    المقارنة: POSIX القياسي مقابل Bashisms
    الميزة POSIX (قياسي) Bashism (خاص بـ Bash)
    المقارنة [ "$A" = "$B" ] [[ "$A" == "$B" ]]
    حلقة رقمية for i in $(seq 1 10) for ((i=0; i<10; i++))
    العمليات الحسابية $(( 1 + 1 )) أو expr $(( 1 + 1 ))
    التوافقية ✅ يعمل على كل الأنظمة ❌ يعمل على Bash فقط
    ⚠️ ما خطورة ذلك في الأنظمة المدمجة Embedded Linux؟

    في الأنظمة المدمجة (مثل أجهزة التوجيه، كاميرات الشبكات، لوحات المراقبة) تكون مساحة التخزين والذاكرة محدودة جداً. لذلك الشركات لا تقوم بتثبيت الـ Bash الثقيل، بل تستخدم برامج Shell خفيفة جداً (مثل ash أو dash داخل BusyBox).

    هذه البرامج الخفيفة تلتزم بشكل صارم جداً بمعيار POSIX لتقليل الحجم.

    إذا قمت بكتابة سكربت يعتمد على مزايا bash الخاصة (مثل for ((i=0; i<10; i++))) وشغلته على اللوحة المدمجة أو أرسلته للعميل — السكربت سيفشل تماماً بـ Syntax Error ولن يتعرف النظام عليه.

    ✅ ماذا يجب أن تتذكر؟

    كمهندس أنظمة مدمجة، الأساس هو قابلية النقل Portability — يجب أن تكتب سكربتات تتوافق حصرياً مع معيار POSIX. استخدم test و [ ] و expr لتضمن عمل كودك الخاص على أي بوردة Board كان في العالم، مهما كان حجمها أو نوع الـ Shell المثبت عليها.

    💡 مراجعة سريعة
    • bash: ثقيل، غني بالمیزات الإضافية، غير مناسب بشكل افتراضي للأنظمة المدمجة.
    • sh (POSIX): قياسي، خفيف، متوفر دائماً في كل الأنظمة، التزم به دائماً.

    المفاهيم الرئيسية

    ioctl هو نداء نظام خاص لإرسال أوامر مخصصة لبرامج تعريف الكيرنل Drivers، ويعتمد على معرفات IDs متفق عليها.
    Kernel Headers هي ملفات .h تحتوي تعريفات الثوابت والأرقام التي تضمن توافق التطبيق مع الكيرنل.
    الـ if في الشل لا تقارن أرقاماً بل تشغل أمراً وتراقب حالة الخروج Exit Status — 0 يعني نجاح True.
    أمر test والقوس [ ] هما نفس البرنامج — أداة للمقارنة وفحص الملفات ترجع 0 أو 1.
    المسافات حول [ ] إجبارية لأنه برنامج تنفيذي وليس جزءاً من قواعد الشل.
    علامات الاقتباس المزدوجة حول المتغيرات في [ ] ضرورية لتجنب خطأ Syntax error عند قيم فارغة.
    Bashisms هي مزايا خاصة بـ Bash غير موجودة في معيار POSIX وتسبب فشل السكربتات في الأنظمة المدمجة.
    الأنظمة المدمجة تستخدم ash / dash داخل BusyBox — خفيفة وتلتزم بـ POSIX صارماً.

    التعريفات المهمة

    ioctl (Input/Output Control): نداء نظام خاص لإرسال أوامر مخصصة لبرامج تعريف الكيرنل.
    Kernel Headers: ملفات ترويسة .h مكتوبة بلغة C تُستخرج من الكيرنل وتُتاح لمساحة المستخدم.
    Exit Status: القيمة الراجعة من أي أمر — 0 نجاح، أي قيمة أخرى فشل.
    Wildcard (*): رمز في الشل يعني "كل شيء" ويُستخدم لمطابقة أسماء الملفات.
    Bashisms: مزايا إضافية في Bash ليست جزءاً من معيار POSIX.
    POSIX: معيار قياسي يضمن قابلية النقل Portability للسكربتات عبر مختلف الأنظمة.

    أخطاء شائعة

    كتابة if VAR == 1 — الـ if تنتظر أمراً وليس مقارنة رياضية مباشرة.
    كتابة [$A -eq $B] بدون مسافات — الشل لن يتعرف على الأمر.
    نسيان علامات الاقتباس المزدوجة: [ -z $NAME ] بدلاً من [ -z "$NAME" ].
    استخدام Bashisms مثل [[ ]] أو for (( )) في سكربتات الأنظمة المدمجة.

    ملاحظات المقابلة الوظيفية

    سؤال: كيف تتواصل تطبيقات الـ User Space مع الـ Kernel Drivers للتحكم المخصص في العتاد؟
    الجواب: عبر نداء النظام ioctl الذي يرسل معرفات IDs محددة معرّفة في Kernel Headers.
    سؤال: لماذا الـ if في الشل مختلفة عن C؟
    الجواب: لأن الشل يقيّم الأوامر لا التعبيرات — يتحقق من Exit Status وليس من القيمة المنطقية.
    سؤال: ما الفرق بين [ و [[؟
    الجواب: [ هو أمر POSIX قياسي، بينما [[ هو Bashism خاص ولا يعمل على الأنظمة المدمجة.
    سؤال: لماذا لا نستخدم Bash في الأنظمة المدمجة؟
    الجواب: لأنه ثقيل على الذاكرة والتخزين، والبديل هو ash / dash داخل BusyBox والخفيف.

    نصائح للحفظ

    fi = if معكوسة: تذكر أن إغلاق الـ if هو كتابتها بالعكس — تماماً كيف أن done يغلق do.
    المسافات حول [ ]: تذكّر أن [ هو "اسم برنامج" — تحتاج مسافة بعد الاسم كما تفعل مع أي أمر.
    الاقتباس المزدوج: "المتغير الفارغ = كارثة بدون علامات اقتباس" — هذه القاعدة تحميك من 99% من أخطاء المبتدئين.
    POSIX = الأمان: في الأنظمة المدمجة، اكتب كما لو أن Bash غير موجود — استخدم فقط ما يفهمه sh.

    نصائح الامتحان

    يُختبر كثيراً: الفرق بين test و [ (الجواب: هما نفس البرنامج).
    يُختبر كثيراً: لماذا المسافات حول [ ] إجبارية (الجواب: لأنه أمر وليس syntax).
    يُختبر كثيراً: جدول خيارات test (-z، -n، -eq، -lt، -gt، -d، -f).
    يُختبر كثيراً: تعريف Bashisms وخطرها في Embedded Linux.

    قائمة المراجعة — Revision Checklist

    • أفهم كيف يعمل ioctl وما علاقته بـ Kernel Headers
    • أستطيع شرح الفرق بين if في الشل و if في C
    • أعرف أن Exit Status 0 يعني نجاح وأي رقم آخر يعني فشل
    • أحفظ الهيكل الكامل لـ if...then...elif...else...fi
    • أفهم قاعدة الفاصلة المنقوطة عند كتابة if و then في سطر واحد
    • أعرف أن [ هو أمر تنفيذي وليس جزءاً من Syntax
    • أحفظ جدول خيارات test كاملاً (-z، -n، =، -eq، -lt، -gt، -d، -f)
    • أفهم لماذا المسافات حول [ ] إجبارية تقنياً
    • أفهم لماذا علامات الاقتباس المزدوجة حول المتغيرات ضرورية
    • أستطيع كتابة حلقة for بثلاث طرق مختلفة لإنشاء القائمة
    • أفهم كيف يعمل Wildcard * مع for
    • أستطيع كتابة حلقة while وحلقة لا نهائية باستخدام true
    • أعرف طريقتي العمليات الحسابية في الشل: $(( )) و expr
    • أفهم الفرق بين POSIX و Bashisms وأمثل على كل منهما
    • أفهم لماذا الأنظمة المدمجة تستخدم ash / dash بدلاً من Bash
    • ألتزم بمعيار POSIX في جميع سكربتاتي
